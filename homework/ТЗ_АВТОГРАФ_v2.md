## 1. Общая информация о проекте

### 1.1 Описание контекста

Проект разрабатывается для проектных институтов и строительных компаний Республики Беларусь, работающих в сфере проектирования и строительства объектов капитального строительства.

Ассистент предназначен для:
- **Главных инженеров проектов (ГИП)** — для контроля соответствия строительства проектным решениям и защиты позиции проектировщика перед заказчиком
- **Специалистов авторского надзора** проектных организаций — для ведения Журнала авторского надзора непосредственно на строительной площадке
- **Специалистов технического надзора** Заказчика — для контроля качества работ подрядчиков

Система поддерживает как локальные точечные запросы (конкретные числовые допуски, требования к материалам), так и глобальные аналитические запросы (сводные правила комплексных процедур контроля по всей нормативной базе).

### 1.2 Проблема, которую решаем

На строительной площадке регулярно возникают спорные ситуации между подрядчиком (строителями) и проектировщиком / техническим надзором. Разрешение споров (замена материалов, отказ от подписания актов скрытых работ, фиксация нарушений) требует **точного и быстрого знания** актуальных Строительных норм (СН), СТБ и постановлений Совмина РБ.

**Ключевые проблемы:**
1. Нормативная база РБ перекрёстно связана — ответ на один вопрос требует одновременного учёта технологических регламентов (СН), организационных процедур, охраны труда и юридических норм
2. Классический RAG не справляется с извлечением связей между изолированными документами
3. Ручной поиск нужного пункта занимает 20–40 минут
4. Технические и юридические ошибки при оформлении документации → штрафы Госстройнадзора, переделки, риск лишения аттестата специалиста

## 2. Цели и задачи проекта

### 2.1 Главная цель

Разработать и ввести в эксплуатацию MVP ИИ-ассистента на основе технологии **LightRAG (GraphRAG)**, сочетающей векторный поиск и граф знаний. Система обеспечивает высокую точность ответов на сложные перекрёстные технические и юридические вопросы путём извлечения связей между нормативными документами РБ.

### 2.2 Измеримые показатели успеха (KPI)

| Метрика | Целевое значение | Фактическое (MVP) |
|---|---|---|
| Скорость ответа (end-to-end) | ≤ 20 секунд | 2–5 секунд (LightRAG) + UI |
| Точность ответов (тестовая выборка) | ≥ 90% корректных со ссылками на НПА РБ | Подтверждено тестовыми кейсами |
| Снижение нагрузки ГИП | −30% времени на типовые консультации | Оценочно достигнуто |
| Доступность системы | ≥ 99% | Обеспечено Docker + restart policy |

### 2.3 Задачи проекта

1. Сформировать базу знаний из нормативных документов РБ (СН, СТБ, инструкции, кодексы)
2. Настроить LightRAG с граф-индексом для перекрёстного поиска между документами
3. Разработать backend API на Python (без внешних фреймворков)
4. Разработать адаптивный веб-интерфейс для работы на стройплощадке
5. Настроить системный промпт с жёсткими ограничениями на использование только норм РБ
6. Обеспечить сохранение истории диалогов в SQLite
7. Развернуть приложение в Docker

|---|---|---|---|---|

| 6 | СН 1.03.01-2019 Организация строительного производства (общие) | PDF | ~60 стр. | ✅ Загружен |

### 5.2 Требования к данным

**Конфиденциальность:** Документы НПА являются публичными. Запросы пользователей (объекты, дефекты) не должны использоваться для обучения базовых LLM-моделей (opt-out обязателен).

**Актуальность:** Периодическое обновление базы при вступлении в силу изменений в строительные нормы РБ — ответственность администратора.

**Доступ к редактированию:** Только администратор (разработчик) имеет доступ к добавлению и удалению документов в LightRAG.

**Форматы:** TXT, MD, PDF, DOCX (PDF и DOCX обрабатываются с предварительной очисткой от колонтитулов и штампов).

## 6. Технические требования

### 6.1 Архитектура системы

```
┌────────────────────────────────────────────────────┐
│              Веб-браузер (Frontend)                │
│     HTML5 / CSS3 / Vanilla JS (адаптив)            │
└───────────────────┬────────────────────────────────┘
                    │ HTTP REST API
┌───────────────────▼────────────────────────────────┐
│           АВТОГРАФ Backend (Python 3.12+)          │
│     Clean Architecture + Domain-Driven Design      │
│  ┌──────────────────────────────────────────────┐  │
│  │  Presentation: ThreadingHTTPServer           │  │
│  │  Application: ChatService                    │  │
│  │  Infrastructure: SQLite + LightRAG Client    │  │
│  │  Domain: ChatMessage, Reference, Answer      │  │
│  └──────────────────────────────────────────────┘  │
└───────────────────┬────────────────────────────────┘
                    │ HTTP
        ┌───────────▼──────────────┐
        │  LightRAG Server (RAG)   │
        │  GraphRAG + Vector Index │
        │  База знаний: 125+ НПА   │
        └──────────────────────────┘
```

### 6.2 Технологический стек

| Компонент | Технология | Обоснование |
|---|---|---|
| **Backend** | Python 3.12+, stdlib only | Нулевые зависимости, быстрый старт |
| **Frontend** | Vanilla JS, HTML5, CSS3 | Без сборки, мгновенная загрузка |
| **Хранение истории** | SQLite | Встроен в Python, ACID, файловый бэкап |
| **RAG Framework** | LightRAG (GraphRAG) | Поддержка перекрёстных связей между НПА |
| **LLM** | OpenAI GPT-4o (через LightRAG) | Высокое качество извлечения сущностей |
| **Граф знаний** | NetworkX / встроенный LightRAG | Двухранговый поиск: local + global |
| **Векторная БД** | Встроенный индекс LightRAG | Хранение эмбеддингов чанков и сущностей |
| **Контейнеризация** | Docker, docker-compose | Воспроизводимое развёртывание |
| **CI/CD** | GitHub Actions | Автоматический lint при PR |

### 6.3 API

| Метод | Эндпоинт | Назначение |
|---|---|---|
| POST | `/api/chat` | Отправка вопроса, получение ответа |
| GET | `/api/health` | Проверка доступности |
| GET | `/api/sessions` | Список последних сессий |
| GET | `/api/documents` | Список документов в базе знаний |
| GET | `/api/documents/pipeline_status` | Статус обработки документов |

### 6.4 Нефункциональные требования

| Требование | Значение |
|---|---|
| **Время ответа** | ≤ 20 секунд (таймаут LightRAG — 35 сек.) |
| **Доступность** | ≥ 99% (Docker restart: unless-stopped) |
| **Язык интерфейса** | Русский (специфика белорусской строительной терминологии) |
| **Безопасность** | API ключи в .env, не коммитить в репозиторий; HTTPS в продакшене |
| **Масштабируемость** | MVP: SQLite + ThreadingHTTPServer; при нагрузке >10 RPS — PostgreSQL + ASGI |

## 7. Ограничения и риски

### 7.1 Ограничения проекта

- **Сроки:** MVP разработан за 4 недели в рамках курсового проекта
- **Команда:** 1 разработчик (full-stack)
- **Бюджет:** Минимальный (использование облачного LightRAG сервера)
- **Масштаб MVP:** До 10 одновременных пользователей

### 7.2 Риски и митигация

| Риск | Вероятность | Митигация | Статус |
|---|---|---|---|
| Галлюцинации LLM (ссылки на нормы РФ вместо РБ) | Высокая | Жёсткий системный промпт: запрет использования знаний вне RAG; явный запрет норм РФ | ✅ Реализовано |
| Ошибки парсинга PDF (колонки, штампы, сканы) | Средняя | Предварительная ручная очистка эталонных документов | ✅ Реализовано |
| Несоответствие номеров пунктов (старые/новые редакции) | Средняя | Hybrid Search (BM25 + Vector) для точного поиска артикулов | ✅ В LightRAG |
| Недоступность LightRAG сервера | Низкая | Health check + корректные сообщения об ошибках пользователю | ✅ Реализовано |
| Переполнение контекста при большой истории | Низкая | Ограничение истории: передаётся максимум 4 последних обмена | ✅ Реализовано |
